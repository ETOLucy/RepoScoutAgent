如何使用 RepoScout

描述需求

写清目标、必须满足的能力、部署限制和可接受的替代方案。例如：寻找可自托管的家庭照片方案，要求人脸识别、手机自动备份和 Docker 部署。

选择上下文模式

  • 自动：系统判断本句是新需求还是补充条件。
  • 新任务：明确丢弃上一轮主题，开始独立研究。
  • 补充条件：在上一轮需求上增加或修改约束。

协作模式与执行过程

协作模式会先给出通俗的目标和验证条件。你可以确认、输入修改意见，或跳过后续确认。搜索后仍可继续补充条件。页面展示可审计的阶段状态和耗时，不展示模型的私有思维链。

什么时候开启 Deep Code

需要了解陌生仓库“代码整体在做什么”、入口在哪里、核心模块如何分工时开启。普通选型先关闭，能更快得到仓库、组合方案和证据矩阵。

  • 只分析最终排名靠前的候选，不扫描所有召回仓库。
  • 按仓库规模自适应抽样：小仓库读得更广，大仓库聚焦入口、repo map 和核心模块。
  • 会增加 GitHub 请求、模型调用和等待时间；结果中的文件数与限制说明代表实际分析边界。
  • 它解释代码职责，不执行候选代码，也不证明项目能够构建或正确运行。

完整搜索与快速降级

默认关闭“快速降级”：系统会等待模型生成完整任务契约；模型解析失败或超过 60 秒时明确停止，不会偷偷改成关键词搜索。只有更在意尽快返回、能够接受需求理解变粗时才开启该选项。

如何看结果

方案组件展示可共同落地的仓库角色；证据矩阵逐项标记 satisfied、violated 或 unknown；Deep Code 区域展示入口、模块职责、源码依据和分析限制。

没有单仓库满足全部条件时，系统会保留最接近的候选，并尝试用独立外部组件补齐方案。关键结论仍应按引用链接和 commit 版本复核。
