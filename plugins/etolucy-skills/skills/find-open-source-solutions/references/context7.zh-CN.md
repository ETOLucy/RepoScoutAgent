# Context7 检索指南

使用 Context7 对最终候选仓库的代码和官方文档进行语义检索。Context7 用于理解已知候选，
不能替代 GitHub 实时发现，也不能保证收录每一个 GitHub 仓库。

## 工具调用顺序

1. 使用准确的 `owner/repository`、项目正式名称和已确认的用户目标调用 Context7 项目解析工具。
2. 选择源仓库和项目身份都匹配的结果，不能只根据相似的显示名称选择。
3. 使用 Context7 文档查询工具提出范围明确、能够直接回答的问题。
4. 保留 Context7 返回的有效来源链接，供最终答案引用。

宿主可能将工具暴露为带 MCP 命名空间的 `resolve-library-id` 和 `query-docs`。使用 Skill
依赖提供的 Context7 工具，不要自行猜测 HTTP 接口。

## 查询设计

只查询做出选择所需的信息。常用问题包括：

- `这个项目的主要产品定位和目标用户是什么？`
- `官方安装文档如何使用 Docker 部署？`
- `需求 X 在哪里被文档说明或由代码实现？`
- `生产部署需要哪些服务、数据库或消息队列？`
- `哪个官方集成点支持协议或 API X？`
- `哪些限制、迁移要求或破坏性变更会影响当前场景？`

对于多仓库组合，应分别查询接口两端。例如，必须分别获得提供方和使用方的 S3 支持
依据，才能判断二者可以通过 S3 配合。

## 未收录时的降级方式

解析失败时：

1. 使用项目正式名称和准确 GitHub URL 重试一次。
2. 优先使用只读 GitHub MCP 的 `search_code` 和 `get_file_contents`，读取 README、官方文档、
   Release 和最小范围的相关源码；不可用时再使用其他 GitHub 或浏览器工具。
3. 正常表达结论，但不要声称该结论经过 Context7 检索。

Context7 服务不可用时，继续使用实时 GitHub 数据和官方来源。只有当降级明显影响推荐时，
才需要向用户说明。

## 检索纪律

- 不要把完整 Context7 响应倾倒到最终答案中。
- 不要用一个巨大问题同时查询所有候选和条件。
- 不要仅凭依赖名称推断运行时能力。
- 部署和兼容性等时效性结论优先使用当前官方文档。
- 代码上下文用于理解架构和实现，不代替产品判断。
- GitHub 仓库和代码搜索是查询驱动的关键词检索，不要将其描述为语义检索。
